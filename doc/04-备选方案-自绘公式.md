# 备选方案：仿 “MathType 式” 自绘公式（不依赖 OMML）

## 为什么需要备选方案

如果在某些宿主/版本里：

- `insertOoxml` 对 OMML/OOXML 的兼容性不稳定，或
- 你希望得到更一致的视觉效果（完全自定义渲染），

可以考虑“自绘公式”路线：**将 AsciiMath 渲染为 SVG/图片插入 Word**，并通过内容控件实现更新。

这条路线的核心 trade-off 是：

- 优点：渲染一致、实现可控、跨宿主更稳定（通常图片插入更稳）
- 缺点：插入的是“图片/图形”，**不是 Word 原生公式**（无法像 OMML 那样在 Word 里逐字编辑）

## MathType 在 Word 中的大体形态（高层理解）

MathType 在 Word 中经常表现为一种“独立对象”，并不是 Word 自带公式编辑器的同一种对象。

从工程角度推断（并结合公开资料的常识性描述）：

- 它更接近 **嵌入对象（如 OLE）/自有数据 + 渲染输出** 的模型
- Word 负责把它当作对象容器承载、排版；MathType 负责编辑与渲染

在 Office.js 环境里，我们通常 **无法**像传统桌面 COM 插件那样直接创建任意 OLE 对象并嵌入，因此“完全仿造 MathType”并不可行。

## 可行的 Office.js 自绘实现

### 方案 A：SVG（推荐）

1. AsciiMath → LaTeX（`asciimath-parser`）
2. LaTeX → SVG（KaTeX 支持输出 SVG，或用 MathJax）
3. 将 SVG 转 base64
4. 使用 Word API 插入图片（如 `insertInlinePictureFromBase64`）
5. 用 `ContentControl(tag=wordmath)` 包裹该图片
6. 将 AsciiMath 源码写入“隐藏元数据”（与当前 OMML 方案一致）

更新：选中对象 → 读取元数据 → 重新渲染 SVG → 替换图片。

### 方案 B：Canvas/PNG

与 SVG 类似，只是渲染为 PNG 再插入。适配性更广，但缩放清晰度不如 SVG。

## 建议的路线策略（双轨）

建议项目采取双轨策略：

- 主路线：**OMML + insertOoxml**（得到 Word 原生公式）
- 备选路线：**SVG 图片**（当 OMML 插入失败/不兼容时自动降级）

UI 上可以提供：

- 插入模式：`原生公式（推荐）` / `图片（兼容模式）`
- 当原生模式报错时，提示可切换到兼容模式
