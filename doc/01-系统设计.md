# 系统设计（MVP 0.1.0）

## 总览

这是一个 Word JavaScript API（Office.js）任务窗格插件，核心链路如下：

1. **AsciiMath**（用户输入）
2. `asciimath-parser`：AsciiMath → **LaTeX**
3. `katex`：LaTeX → **MathML**
4. `mathml2omml`：MathML → **OMML**
5. 组装最小 **OOXML**（包含 OMML）
6. `Word.Range.insertOoxml()`：插入/更新到文档

## 为什么用 OMML + OOXML

- Word 的“原生公式”底层是 OMML（Office Math Markup Language）
- Office.js 目前没有“直接创建数学对象”的高级 API，因此用 `insertOoxml` 插入包含 OMML 的 OOXML 是更通用的方式
- 插入后在 Word 里是标准公式对象，利于排版/复制/粘贴

## 插入与更新的定位方案

### 内容控件（Content Control）

每次插入时，会在光标处创建一个 `ContentControl`，用于标识“这是本插件管理的公式”，并便于后续更新：

- `tag = "wordmath"`：用于识别
- `appearance = BoundingBox`：在 Word 内可见边框（便于用户知道它是一个可管理对象）

更新时逻辑为：

- 若当前选区在 `tag = "wordmath"` 的内容控件内，则“更新”
- 否则创建新的内容控件并“插入”

### AsciiMath 源码的持久化（自包含元数据）

OMML 本身无法可靠地逆向还原为 AsciiMath，因此需要把“源码”存起来。

当前版本采用 **随公式内容自包含的隐藏元数据**（OOXML 隐藏文本）：

- 插入 OOXML 时，在公式段落里追加一个带 `w:vanish` 的隐藏 run
- 该 run 的文本包含 `WMASCII:<base64>`，base64 内容为 UTF-8 的 AsciiMath

优点：

- **复制/粘贴** 公式时，元数据会一起复制，仍可“从选中公式加载”
- 不依赖 `contentControlId` 等易变标识
- 不污染文档可见内容（除非用户开启“显示隐藏文字”）

限制：

- 如果用户开启 Word 的“显示隐藏文字”，会看到一段隐藏标记（可在后续版本提供开关，或改用更隐蔽的元数据载体）

## 代码结构

- `src/taskpane/*`：Vue3 UI（输入、预览、按钮、错误显示）
- `src/core/convert.ts`：转换（AsciiMath→LaTeX→MathML→OMML）
- `src/core/ooxml.ts`：OMML → OOXML 包装
- `src/core/word.ts`：Word 插入/更新、从选区读取
- `src/commands/commands.ts`：Ribbon 按钮命令入口（目前仅用于 ShowTaskpane）
- `manifest/wordmath.xml`：插件 Manifest

## 错误处理策略

- UI 层捕获所有异常并展示到任务窗格
- 转换链路任一步为空/失败均抛出可读错误
- 当不在 Word 环境中运行时直接报错提示
